快照只是一个时间冻结点，不能替代 Veeam / NBU 等专业备份工具。长期保留快照 = 给自己埋雷。

快照到底是什么？

创建快照时，VMware 做了三件事：

  1. 原始磁盘变为 只读，创建 delta 文件（-000001.vmdk）
  2. 所有新写入 全部进入 delta 文件
  3. delta 文件 只增不减，VM 内部删文件也不会缩小
base.vmdk (200GB, 只读)
  └── delta-000001.vmdk (快照1后的写入)
      └── delta-000002.vmdk (快照2后的写入)
          └── delta-000003.vmdk (当前写入点)
删除快照 = 合并 delta 文件回父磁盘。这个合并过程需要大量 I/O 和临时空间，是所有问题的根源。

坑一：快照膨胀

真实翻车 SQL Server 快照忘了删

给一台 SQL Server VM 打了快照，"临时"用一下。3 个月后：
base.vmdk = 200GB（只读没变）
delta = 180GB（3 个月的写入全在这里）
数据存储从 60% 飙到 95%，触发告警。删除预计 8+ 小时。

膨胀速度取决于什么？

因素影响
VM 写入量数据库、日志密集型 VM 膨胀最快
存活时间快照放 3 天和 3 个月，天差地别
快照层数多层叠加，每层独立膨胀
磁盘类型Thin Provisioning 的 VM 更难发现膨胀

膨胀的 5 大后果

  • 存储爆满 — 同一 datastore 上所有 VM 同时暂停
  • 删除巨慢 — 180GB delta 合并要几小时到十几小时
  • 性能骤降 — 合并期间 I/O 被吃掉，VM 明显卡顿
  • 合并失败 — 空间不足时合并失败，可能丢数据
  • vMotion 失败 — 带大快照的 VM 迁不动

坑二：快照删除卡住

卡在 0% 通常是存储空间不足

合并需要临时空间（约等于 delta 大小），空间不够就卡死。
处理：先腾空间，再重试删除。

卡在 95% 数据已合并，卡在元数据更新

数据合并完成了，最后一步重命名文件卡住。
处理：不要取消任务！耐心等待。超过 24 小时再干预。

最危险 显示"无快照"但 delta 文件还在

快照管理器空了，但文件系统里还有 -000001.vmdk。元数据与文件不一致。
处理：VM → Snapshots → Consolidate

锁冲突 报错 "File is locked"

其他主机持有 SCSI 磁盘锁，合并无法获取独占访问。
处理：vmkfstools -D 查锁，找到持有锁的主机 MAC，vMotion 后重试。

6 大故障场景处理方案

场景一：存储空间不足导致合并卡住

症状：任务卡 0%，datastore > 95%

处理：腾空间 → 确保可用空间 > 最大 delta → 重试删除

# SSH 到 ESXi，检查空间
df -h /vmfs/volumes/datastore1/

# 找最大的 delta 文件
find /vmfs/volumes/datastore1/ -name "*delta.vmdk" \
  -exec ls -lh {} \; | sort -k5 -hr | head -20

场景二：SCSI 磁盘锁冲突

症状：日志出现 "File is locked"

处理：查锁 → 找持有者 → vMotion 后重试

# 检查谁持有磁盘锁
vmkfstools -D /vmfs/volumes/ds/vm/vm-000001-delta.vmdk

# 输出中的 MAC 地址 = 持有锁的主机
# vMotion 到其他主机后重试

场景三：CBT（Changed Block Tracking）损坏

症状：Cannot open the disk

处理：关机 → 禁用 CBT → 删除 CTK 文件 → 开机 → 重试

# PowerCLI 禁用 CBT
$vm = Get-VM "vm-name"
$spec = New-Object VMware.Vim.VirtualMachineConfigSpec
$spec.changeTrackingEnabled = $false
$vm.ExtensionData.ReconfigVM($spec)

场景四：快照链断裂（元数据不一致）

症状：快照管理器为空，但 delta 文件还在

处理：先尝试 Consolidate → 失败则关机后 clone 重建

# 关机后 clone 重建磁盘
vmkfstools -i vm-000001.vmdk vm-consolidated.vmdk -d thin

# 修改 .vmx 指向新磁盘，启动验证

场景五：VSS 快照失败（Windows VM）

症状：quiesce 快照创建/删除失败

处理：VM 内部修复 VSS 服务

# 在 Windows VM 内部执行
vssadmin list writers     # 找不正常的 Writer
Restart-Service "VMware Snapshot Provider" -Force
Restart-Service "VSS" -Force
vssadmin delete shadows /all /quiet

场景六：合并超时（存储 I/O 慢）

症状：任务长时间无进展，存储延迟高

处理：esxtop 查 KAVG（> 30ms 需关注），安排低峰期重试

PowerCLI 快照管理脚本

快照健康检查

# 找出所有超过 24 小时的快照，按大小排序
Get-VM | Get-Snapshot | Where-Object {
    $_.Created -lt (Get-Date).AddHours(-24)
} | Select-Object VM, Name, Created,
    @{N='AgeDays';E={[math]::Round(((Get-Date)-$_.Created).TotalDays,1)}},
    @{N='SizeGB';E={[math]::Round($_.SizeGB,2)}} |
  Sort-Object SizeGB -Descending | Format-Table -AutoSize

检查需要 Consolidate 的 VM

Get-VM | Where-Object {
    $_.ExtensionData.Runtime.ConsolidationNeeded
} | Select-Object Name, PowerState

批量清理过期快照

$cutoff = (Get-Date).AddDays(-3)
$expired = Get-VM | Get-Snapshot | Where-Object { $_.Created -lt $cutoff }

# 预览
$expired | Select-Object VM, Name, Created, SizeGB | Format-Table

# 确认后执行删除
# $expired | Remove-Snapshot -Confirm:$false

紧急处理 Checklist

存储告急（> 90%）

  • 找出最大的快照：Get-VM -Datastore ds | Get-Snapshot | Sort SizeGB -Desc
  • 优先删测试环境的快照
  • 空间不够删？迁移小 VM 到其他存储
  • 确保可用空间 > 最大 delta 后再删
  • 从最新的快照开始删（减少合并量）

快照删除卡住

  • 不要取消正在进行的任务！
  • 检查存储空间是否足够
  • SSH 查日志：grep snapshot /var/log/hostd.log | tail -20
  • 检查磁盘锁：vmkfstools -D <delta-file>
  • 尝试 Consolidate
  • 以上都不行 → 关机后 clone 重建
  • 最终方案 → 联系 VMware 支持（vm-support 收集诊断包）
